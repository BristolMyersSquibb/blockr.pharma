Patient Profile — making the visualization sidebar smaller

The sidebar is a fixed 300 px and each card spends ~60 px of height (40 px icon + title + two-line description). On a laptop inside the dock, that is roughly a quarter of the block's width spent on a picker you touch once per session. Below are five alternatives, from a pure CSS diet to removing the sidebar entirely. All mockups use the real viz metadata (labels, domains, icons, colors) and are lightly interactive — click cards, chips and buttons.

Current design reference — 300 px

What ships today: header + search + SELECTED / AVAILABLE card list. The description line and the 40 px icon are what make it wide and tall.

1 · Compact rows 300 → 232 px · rows 60 → 29 px

Keep the sidebar and its whole interaction model, but put it on a diet: one-line rows, 22 px icon chips, description demoted to a native tooltip (hover a row). Search, sections, domain groups and drag-reorder all survive unchanged. Twice as many vizs visible without scrolling.

Visualizations

Interactive: click rows to select / deselect; hover for the description tooltip.

Pros

  • Cheapest change by far — CSS + dropping one <p> from build_card()
  • Nothing about search / drag / sections / sync logic changes
  • ~2× vertical density; whole AVAILABLE list often fits without scrolling

Cons

  • Still a permanent 232 px column
  • Descriptions only on hover (fine — they are mostly PARAMCD lists)
Touchpoints: patient-profile.css (.pp-sidebar width, .pp-card-* sizes) and build_card() in patient-profile-block.R (move description to title= attribute).

2 · Icon rail with hover flyout 300 → 52 px

Collapse the sidebar to a 52 px rail of icon buttons — selected vizs on top (accent ring + left bar), a divider, then the available ones. Hovering the rail slides out a flyout with the full labeled list (search included); it retracts when the pointer leaves. The picker stays permanently visible and one click away, at a sixth of the width.

Pin ▸ to keep open · Esc to close

Interactive: hover the rail to open the flyout; click icons or rows to toggle.

Pros

  • Selection state always visible at a glance (colored icons + accent marks)
  • ~250 px returned to charts, permanently
  • Familiar pattern (VS Code activity bar, dock sidebars)

Cons

  • Icons alone are ambiguous — 5 lab panels share the droplet icon, so the flyout is mandatory
  • More JS: flyout open/close, hover intent, pinning
  • Drag-reorder needs to live in the flyout, not the rail
The collapse/expand plumbing (pp-pin-btn, pp-expand-btn) already exists — the rail replaces the fully-collapsed state rather than adding a third mode.

3 · Chip bar on top sidebar gone · ~36 px of height

Drop the sidebar entirely. Selected vizs become removable chips in a bar above the charts, sharing the row with an + Add button that opens the searchable card list as a dropdown. Chips drag to reorder; the domain color dot keeps them scannable. Horizontal cost: zero.

Interactive: remove chips with ×, click + Add to open the picker menu.

Pros

  • Zero horizontal cost — charts get the full block width
  • Selected set and its order are always visible (chips are the order)
  • Chips double as quick-remove; matches the existing per-panel × button

Cons

  • AVAILABLE list only visible inside the dropdown — less browsable/discoverable
  • Many selected vizs wrap the bar to 2 rows, eating the height it saved
  • Bigger refactor: sidebar render, search JS and toggle sync move into a menu
The dropdown should reuse build_card() as-is; portal it to <body> like Blockr.Select so it escapes .pp-chart-area overflow (same clipping trap as the gear popover).

4 · Toolbar popover picker sidebar gone · 0 px standing cost

The most radical: the whole picker lives behind one toolbar button next to the subject picker, showing a count badge. Clicking opens the current sidebar — cards, search, SELECTED / AVAILABLE — as a popover, unchanged. When closed, the profile is 100% charts.

Interactive: click Visualizations to open/close the picker; toggle cards inside — the badge updates.

Pros

  • Absolute maximum chart space — nothing persistent at all
  • Card list, search and section logic reused verbatim inside the popover
  • Matches how the gear popover already works — consistent block chrome

Cons

  • Selection state invisible while closed (only a count)
  • Two clicks to toggle one viz; browsing while comparing charts means reopening
  • First-time discoverability is the weakest of the five
Essentially "move pp-sidebar into a portalled popover". The existing collapse state machine goes away; the popover needs the same <body> portal treatment as Blockr.Select.

5 · Overlay + auto-hide 0 px steady-state · sidebar unchanged

Keep today's sidebar pixel-for-pixel, but change its behavior: it overlays the charts instead of pushing them, opens from a slim 20 px edge strip, and auto-hides after you add or remove a viz. Steady state is all charts; the full browsable list is one hover away. Pin it (📌 already exists) to get exactly today's pushed layout back.

Visualizations

Interactive: click the thin edge strip to slide the sidebar over the charts; selecting a card auto-hides it; click the scrim to dismiss.

Pros

  • Zero visual redesign — same cards, same search, same everything
  • Full-width charts by default, full browsable list on demand
  • Pin restores current behavior for users who prefer it (state already serialized)

Cons

  • Overlay hides the leftmost chart while open — no live side-by-side while picking
  • Auto-hide can feel jumpy when adding several vizs in a row (mitigate: hide only on scrim click / Esc, not on select)
Mostly CSS: .pp-sidebar becomes position:absolute over .pp-layout with a transform transition; the existing collapsed/pin state machine drives it. The 20 px strip replaces today's floating expand button.

Recommendation

Ship 1 + 5 together. Compact rows are nearly free and help even while the sidebar is open; overlay + auto-hide makes the steady state 100% charts without redesigning anything, and the existing pin gives conservative users today's layout back. Combined: a 232 px overlay that costs 20 px when idle.

If you want a stronger visual statement later, 2 (icon rail) is the natural second step — it reuses the compact rows inside its flyout. Options 3/4 only become attractive if the sidebar concept itself should disappear from the block chrome.
